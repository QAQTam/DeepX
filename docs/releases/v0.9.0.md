# DeepX 0.9.0：从桌面程序走向多客户端平台

发布日期：2026-07-22

DeepX 0.9.0 不是一次普通的功能更新。它重新划分了产品的进程边界、代码所有权和发布方式，为 1.0.0 的稳定架构奠定基础。

此前，DeepX 的主要形态是一个 Tauri 桌面应用：SolidJS 界面通过 Tauri command 直接调用 Rust 业务代码，桌面窗口、会话管理、Agent 生命周期和本地数据访问彼此耦合。关闭 GUI、切换会话或更换客户端技术栈，都可能影响正在执行的任务。

0.9.0 将这套结构改造成独立的 Rust 后端与可替换客户端：

```text
deepx-desktop ── Control Protocol ── deepx-daemon ── Agent worker
未来 deepx-tui ────────┘
```

这意味着 Desktop 不再是 DeepX 本身，而只是 DeepX 的一个客户端。会话、Agent、工具和用户数据继续由 Rust 实现，但不再寄居于某个 GUI 进程中。

> 发布历史说明：Git 标签 `v0.9.0` 当前指向 `db8f387`。Daemon 生命周期协调和流式管道稳定化分别在后续提交 `53169b5`、`a87514f`，Electron 对应修复在 `3bb9655`、`07ade94`。这些改动属于当前 0.9 系列的完整稳定状态；对外重新制作安装包时，应将它们纳入新的补丁版本或更新后的发布制品，而不应移动已经公开的 Git 标签。

## 本次版本的核心变化

### 1. 独立的 `deepx-daemon`

新增常驻的 Rust 后端进程 `deepx-daemon`。它成为 `.deepx` 数据、会话状态、配置、Workspace、Git、Plan、Skills、统计信息和 Agent worker 的唯一业务宿主。

- Desktop 关闭后，Daemon 和正在执行的任务继续运行。
- Daemon 采用按用户运行的单实例模型，不注册系统服务，也不监听局域网地址。
- 控制服务只绑定 loopback，并使用每次启动生成的 Bearer token 鉴权。
- Daemon 原子写入 `.deepx/daemon.json`，公布 endpoint、PID、协议版本、epoch、构建身份和可执行文件路径。
- 提供 `run`、`status`、`stop` 以及安全的 idle-only 停止能力。
- Agent worker 继续由同一套 Daemon 可执行文件的内部子命令承载，不产生第三个用户级应用。
- 现有 Session、Turso、Config、Workspace 和 Plan 数据格式保持原地兼容。

### 2. 版本化控制协议

`deepx-proto` 新增面向客户端的 Control Protocol。原有 `Ui2Agent` / `Agent2Ui` 保留为 Daemon 与 Agent worker 的内部语义，不再直接承担 GUI 进程边界。

新协议提供：

- `ClientHello` / `ServerHello` 版本协商；
- 带唯一 `request_id` 的强类型请求与响应；
- 稳定错误码，而不是二次编码的 JSON 字符串；
- `server_epoch`、全局 `seq`、会话序号和 Snapshot；
- 同一 epoch 内的事件续传，以及无法续传时的完整 Snapshot 恢复；
- 消息大小、连接数量、握手时间和请求速率限制；
- `session.*`、`interaction.*`、`config.*`、`workspace.*`、`git.*`、`plan.*`、`skills.*`、`stats.*` 和 `daemon.*` 领域方法。

这一协议是 Desktop、未来 TUI 以及其他可信本地客户端共同依赖的产品边界。

### 3. 会话租约和断线恢复

Daemon 允许多个客户端查看全局会话与活动状态，但同一会话在同一时刻只有一个控制者。

- Attach 会建立绑定 `client_instance_id + seed` 的租约。
- 第二个客户端会收到 `session_busy`，不能读取该会话的正文事件或发送控制命令。
- 客户端每 5 秒发送心跳；断线后保留 15 秒重连窗口。
- 相同客户端身份可以恢复租约，超时后租约自动释放。
- Permission、Ask、Plan 等交互只发送给租约持有者。
- Daemon epoch 改变后，客户端重新 Attach，并从持久化状态和 Snapshot 恢复界面。

多会话不再意味着启动多份后端。一个 Daemon 可以同时管理多个 Session 和对应的 Agent worker。

### 4. 可复用的 `deepx-client`

新增 Rust crate `deepx-client`，集中实现：

- Daemon 发现与自动拉起；
- Bearer 鉴权和协议握手；
- 请求关联、心跳和重连；
- epoch / seq 续传游标；
- 会话 Attach、Detach 与租约恢复；
- 统一事件订阅。

未来的 Ratatui 客户端应直接复用这个 crate，而不是读取 `.deepx` 文件或复制 Daemon 逻辑。

### 5. 独立 Electron Desktop

新增独立仓库 `QAQTam/deepx-desktop`，使用 Electron、SolidJS 和受限的 `contextBridge`：

```text
SolidJS Renderer
       │ 受限 API
Electron Preload
       │ 校验后的 IPC
Electron Main
       │ 带鉴权的 WebSocket
deepx-daemon
```

Renderer 不启用 Node.js，也不能读取 Daemon discovery token。Electron Main 负责 Daemon 发现、隐藏启动、构建身份验证、请求关联、心跳、重连、租约和桌面原生能力。

文件/目录选择、确认框、打开本地路径和外部链接已经迁移到 Electron 原生桥接。原 SolidJS 界面的 Session、工具、权限、Ask、Plan、Skills、Goal、Git、Settings、统计、Markdown、KaTeX 和代码高亮能力均得到保留。

### 6. 可复现的跨仓库打包

后端与前端现在可以独立 Clone、独立测试、独立发布，同时仍能精确组合成一个安装包。

`deepx-desktop` 通过 `deepx-backend.lock.json` 锁定：

- DeepX 后端仓库；
- 产品版本和协议版本；
- 精确 Git commit；
- Release manifest；
- 对应平台的 Daemon artifact 和 SHA-256。

正式打包从 GitHub Release 下载并验证 Daemon；本地联调则可以从相邻的 DeepX 源码仓库构建。两个仓库不使用 Git submodule，版本关系由发布制品和锁文件定义。

### 7. 流式输出和连接稳定化

0.9 系列后期重点修复了“工具调用期间停止流式渲染，直到 Done 才一次性显示”以及高并发会话下误报 `daemon disconnected` 的问题。

- Daemon 将业务请求执行与 WebSocket 写入解耦。
- Response、Heartbeat、Snapshot 和控制错误使用独立的高优先级队列。
- Agent 不再逐字符制造跨进程事件，而是按照约 16 ms 的显示帧进行有界、有序合批。
- Electron 对文本 delta、工具调用预览和终端输出进行有序合批。
- 不同事件类型保持原始顺序，完成事件不会越过尚未发送的文本。
- 消费端严重过载时主动断开并通过 Snapshot 恢复，而不是静默丢失内容。

这不是简单的“让文字显示得更快”，而是重新建立从 Provider SSE、Agent、Daemon、WebSocket、Electron IPC 到 SolidJS reducer 的背压边界。

## 0.9.0 中保留和暂停的能力

本次迁移不改变用户数据格式，也不要求清空 `.deepx`。会话历史、配置、Workspace、Git、Plan、Skills、统计信息和 Turso 镜像仍按原格式工作。

以下内容不属于 0.9.0 的完成条件：

- 不实现正式 TUI，只通过 `deepx-client` API 和连接测试证明可接入性；
- Companion/Pet 暂停启动，源码和独立测试保留，未来再迁入 Daemon；
- 不把 Daemon 注册成 Windows Service；
- 不开放局域网或远程控制端口；
- Linux/macOS 在 0.9 阶段优先保持结构正确和可编译，完整发布体验继续在 1.0 前验证。

## 为什么最终选择 Electron，而不是更轻量的 Tauri

这是 0.9.0 最晚、也最重要的一次技术栈决定。

Tauri 的优势真实存在：安装体积更小、空载内存更低、Rust 集成自然，并且可以使用操作系统自带 WebView。对于页面简单、平台 WebView 表现一致的工具，它仍然是优秀方案。

但 DeepX 的桌面界面已经不是一个简单表单。它长期显示流式 Markdown、语法高亮、Diff、KaTeX、图表、工具时间线、大量增量事件和多会话状态。在实际 Linux 测试中，系统 WebView 的字体、布局、渲染和整体观感与 Windows 存在明显差异。依赖各发行版的 WebKitGTK 版本，也让“同一份前端代码”不等于“同一种用户体验”。

Electron 最终胜出的原因是确定性：

1. **一致的渲染基线。** Electron 随应用提供 Chromium，Windows、Linux 和 macOS 更接近同一个字体、CSS、Canvas 和 DevTools 行为。
2. **更适合当前 UI 复杂度。** Markdown、Shiki、KaTeX、Diff 和图表等前端依赖可以在明确的 Chromium 版本上测试，而不是同时适配多个系统 WebView。
3. **问题更容易复现。** 流式事件、IPC 调度、渲染性能和内存问题可以围绕固定运行时诊断，减少平台特有的偶发差异。
4. **Rust 优势已经位于正确的位置。** DeepX 的安全边界、数据、Agent 和工具仍全部在 `deepx-daemon`。桌面壳不再需要为了“使用 Rust”而选择 Tauri，因为真正重要的 Rust 后端已经独立存在。
5. **前后端分离更彻底。** Electron Main 是一个很薄的适配层，SolidJS Renderer 不接触 Node.js 或业务文件。未来替换 UI、增加 Browser 壳或独立发布前端，不再要求改动 Rust workspace。
6. **维护一条成熟路线胜过维护两个轻量壳。** 同时维持 Tauri 和 Electron 会复制测试、原生桥接、打包和平台问题，并持续产生功能漂移。

我们接受 Electron 更大的安装体积和更高的基础内存占用。这是一次明确的产品取舍：DeepX 选择跨平台表现、诊断能力和发布确定性，而不是追求最小的 GUI 壳。Daemon 与 Agent 的资源管理仍由 Rust 掌控，Electron 的成本只存在于用户选择打开图形界面的时候；未来的 Ratatui 客户端仍可提供真正轻量的终端入口。

## 迈向 1.0.0 的最终结构

1.0.0 将不再是 Tauri + Electron 双轨产品，而是两个职责明确、可独立发布的仓库。

```text
QAQTam/DeepX
├─ deepx-proto       版本化协议与公共类型
├─ deepx-runtime     与界面无关的应用服务
├─ deepx-daemon      数据与 Agent 的唯一宿主
├─ deepx-client      Rust 客户端，供 TUI 等使用
├─ session/config/tools/message/skills/...
└─ 不包含 GUI

QAQTam/deepx-desktop
├─ electron/         Main、Preload、原生能力和 Daemon 生命周期
├─ src/              SolidJS Renderer
├─ runtime adapters  BackendClient 与 Desktop API
├─ backend lock      精确绑定后端发布制品
└─ Windows/Linux/macOS 桌面打包
```

数据流将固定为：

```text
Electron Desktop ─┐
Ratatui TUI ──────┼─ versioned control protocol ─ deepx-daemon ─ Agent workers
未来可信客户端 ──┘
```

0.9 系列的最终收口已经补齐 Electron 的显式 Session Detach、`after_epoch + after_seq` 续传能力，从 Rust workspace 删除了旧 `deepx-tauri`、Tauri/Wry 依赖和过时桥接脚本，并将 `deepx-desktop` 确立为唯一 GUI 和前端事实来源。

1.0.0 之前仍计划完成：

- 完成 Windows 安装、升级、卸载和 Daemon 生命周期验证；
- 完成 Linux/macOS 的 Electron 视觉与打包验收；
- 固化协议兼容政策、Release manifest 和跨仓库 CI；
- 保证任意客户端都不能绕过 Daemon 直接写入业务数据。

## 总结

0.9.0 解决的根本问题不是“把 Tauri 换成 Electron”，而是把 DeepX 从一个 GUI 内部的 Rust 应用，重构成一个拥有稳定协议、常驻运行时和多个潜在客户端的平台。

Electron 是最终桌面实现，Rust Daemon 是产品核心，Control Protocol 是两者之间长期稳定的契约。1.0.0 将以这三个边界为基础，结束桌面技术栈试验期，并开始承诺可维护的兼容性与发布体验。
