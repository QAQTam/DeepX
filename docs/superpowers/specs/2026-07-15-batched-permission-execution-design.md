# 批量权限决策与并行执行设计

## 目标

修复同一模型轮次包含多个需授权工具时，授权响应立即触发串行执行、界面静默且模型恢复时机不明确的问题。新的语义是：先收集本批全部权限决策，再统一执行一次，最后只恢复一次模型 loop。

## 批次状态

每个模型工具调用在当前 `TurnState` 中保留稳定记录：

- `pending`：等待用户决策；
- `approved`：已批准，持有单次授权对象但尚未执行；
- `rejected`：已拒绝，已生成失败工具结果；
- `completed`：执行完成并已写入工具结果。

自动允许的工具同样进入本批待执行集合。如果本轮存在任何待授权项，任何工具都不得提前执行。

## 后端数据流

1. `admit_batch` 对整批工具完成准入，返回自动授权项、待授权项、拒绝项和 ask_user 项。
2. `TurnEngine` 在存在待授权项时保存完整批次并进入 `PermissionPending`。
3. `PermissionResponse` 只消费对应 challenge 并记录 approved/rejected，不执行工具。
4. 仍有 `pending` 时继续等待；最后一项决策完成后，构造唯一执行集合。
5. 使用统一批量执行器：无冲突工具最多四路并行；同一文件的写操作按模型原始顺序串行。
6. stdout/stderr 继续通过现有 `ExecProgress` 事件发送。
7. 全批结果写入消息存储后，发出一次完整工具轮次并进入下一次 Gate 请求。
8. 批次含 ask_user 时，工具执行完成后按既有顺序进入 ask_user；全部交互结束后才恢复模型。

UI 发起的独立工具调用不属于模型批次，保持现有即时执行语义。

## 生命周期与清理

- 重复或未知 permission response 必须忽略，不得重复执行。
- 拒绝、过期授权都生成对应失败工具结果。
- cancel、切换/新建/恢复会话、agent 退出时清空 challenge 与尚未执行的授权对象。
- 单次授权对象只允许从 `approved` 转移到 `completed` 一次。

## 前端交互

- 只保留中央 `PermissionPrompt`，删除未接线的旧 `PermissionDialog` 与旧样式导入。
- 权限队列展示“第 n/m 项”，逐项记录批准或拒绝。
- 最后一项决策完成后关闭弹窗，ChatView 过程区展示等待/执行状态和实时输出。
- 低、中风险批准按钮使用统一中性样式；高风险批准按钮使用红色实心；拒绝始终中性。
- 底层审计状态继续保留，但不在对话内容中显示 `permission: approved`。

## 错误处理

- 某一工具失败不阻止同批其他无冲突工具收集结果。
- worker panic 转换为该工具的失败结果。
- Gate 恢复失败必须发出 error 并结束前端运行态，不能无限保留停止按钮。

## 验证

- 四个需授权 exec：前三次响应后不执行、不恢复；第四次响应后最多四路并行并只恢复一次。
- 混合自动允许、批准、拒绝：授权完成前均不执行，结束后结果完整且顺序可关联。
- 冲突写操作保持串行，非冲突 exec 可并行。
- 重复响应、取消、会话切换不会执行陈旧授权。
- stdout/stderr 在批准后的 exec 执行期间可见。
- 前端队列计数正确，只有高风险批准按钮为红色，旧权限 UI 不再进入构建。

## 非目标

- 本轮不增加“一键批准全部”。
- 不改变权限等级与风险判定规则。
- 不扩大并行上限，不重新设计工具协议之外的聊天界面。
